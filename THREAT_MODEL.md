# EncSend Crypto Core Threat Model

## Scope

This threat model applies to the browser-side cryptographic core published in `@encsend/crypto-core`.

It does not claim to fully model the hosted EncSend product. Product-level controls such as OTP delivery, preview gates, session handling, route protection, server-side policy enforcement, and audit logging live outside this package.

## Security Goals

The current crypto core is intended to provide:

- confidentiality of secret plaintexts and encrypted submissions at rest on the server
- confidentiality of link keys carried in URL fragments
- confidentiality of owner master keys, owner-wrapped object keys, and encrypted key-vault backups
- cryptographic integrity for encrypted payloads and wrapped keys

## Assets Protected by the Core

- secret plaintext
- secure request submission plaintext
- reply plaintext
- public-link fragment keys such as `#k=...` and `#rk=...`
- owner master key material
- recovery profile ciphertext integrity
- key-vault ciphertext integrity

## Data Intentionally Not Hidden by the Core

The crypto core does not hide, and does not claim to hide:

- account identifiers and email addresses
- recipient OTP target addresses
- public IDs and server-side access tokens
- creation timestamps, expiry timestamps, status flags, and view/submission counters
- ciphertext sizes
- route access timing and request frequency
- audit events generated by the product

This is a zero-knowledge design for content and keys, not for all product metadata.

## Trust Assumptions

The current design assumes:

1. the browser receives authentic application JavaScript over a trusted transport such as HTTPS
2. the endpoint is not already compromised by malware, hostile extensions, or a hostile local user
3. the browser Web Crypto implementation behaves correctly
4. users share the full public URL, including the fragment key, only with intended recipients
5. recovery passphrases and account passwords are chosen with sufficient entropy for their purpose

If the server or deployment path can silently replace the client JavaScript, the server can also exfiltrate plaintext before encryption or after decryption. Open-sourcing the crypto core improves reviewability, but it does not remove that deployment trust assumption by itself.

## Adversaries Considered

The current package is designed to resist:

- a storage attacker who can read database rows or object storage ciphertexts
- a network observer who cannot modify the delivered application
- an attacker who obtains only the server-visible part of a shared link and not the fragment key
- an attacker who obtains encrypted recovery artifacts but not the recovery secret

## Adversaries Out of Scope

The current package does not defend against:

- a malicious or compromised origin serving modified JavaScript
- a fully compromised browser or operating system
- browser extensions reading DOM state or decrypted plaintext
- users forwarding full links, screenshots, or copied plaintext
- phishing or account-takeover attacks outside the crypto protocol
- timing, size, or traffic-analysis leakage at the product boundary

## Zero-Knowledge Boundary

Within the intended boundary:

- plaintext is encrypted and decrypted in the browser
- link fragment keys are not sent to the server by normal browser navigation
- owner recovery artifacts can remain server-stored ciphertext

Outside that boundary:

- the service still learns operational metadata
- any server-delivered JavaScript remains part of the trust base
- product-level policy controls are separate from cryptographic confidentiality

## Open-Core Boundary

This package exposes the cryptographic core for public review. It does not, by itself, prove the security of:

- OTP or password challenge orchestration
- one-time-view deletion semantics
- preview-gate enforcement
- access-control and session-management logic
- infrastructure, deployment, or operational monitoring

Those parts must be reviewed separately in the product code or corresponding protocol documentation.
